AWS Client VPN으로 사설 리소스 접근 경로를 정리하면서, 인증과 암호화를 설명하는 내 기준도 다시 정리할 필요가 생겼다.  
실무에서는 종종 "VPN도 붙었고 내부 ALB도 HTTPS니까 인증과 보안은 충분하다"라는 식으로 여러 층의 책임을 한 문장으로 묶어 이해하게 된다.  
그런데 이 문장은 보통 세 가지를 한꺼번에 섞는다.

- 누가 사설망 안으로 들어올 수 있는가
- 상대방이 정말 우리가 신뢰하는 서버/클라이언트인가
- 들어온 뒤 어떤 HTTP 요청을 수행할 수 있는가

이 셋은 겉보기엔 비슷해 보여도, 실제로는 서로 다른 암호화 객체와 서로 다른 증명 방식을 가진다.  
VPN의 mTLS는 X.509 인증서와 개인 키 소유 증명을 통해 **터널을 열 수 있는 주체**를 판단한다.  
반면 HTTP 인증은 이미 열린 연결 위에서 **누가 어떤 메시지를 보냈는지**를 판단한다.

이 글은 그 차이를 기술 문서처럼 다시 정리한 노트다.  
핵심 질문은 세 가지다.

- VPN의 mTLS는 정확히 무엇을 인증하는가
- VPN 터널의 TLS와 내부 HTTPS의 TLS는 무엇을 각각 보호하는가
- HTTP 표준에서 말하는 인증은 어디서 시작되는가

## 문제 정의

사설 리소스 접근 체계가 커지면 아래 두 가지가 쉽게 섞인다.

- 네트워크 경계 안으로 누가 들어올 수 있는가
- 경계 안에 들어온 뒤 어떤 HTTP 요청을 수행할 수 있는가

이 둘을 분리하지 않으면 설계가 빠르게 흐려진다.  
예를 들어 VPN 연결 성공을 곧바로 애플리케이션 인증 성공처럼 취급하면, 실제로는 다음과 같은 문제가 생긴다.

- VPN 접속 성공이 내부 서비스 권한 승인처럼 오해된다
- 내부 HTTP 서비스가 별도 인증 없이 "사설망 안이니까 안전하다"는 가정에 기대게 된다
- 감사 로그 기준이 네트워크 접속과 애플리케이션 행위를 구분하지 못한다
- 오프보딩 시 VPN 인증서 폐기와 서비스 계정 회수가 따로 관리되지 않는다

여기에 암호화 관점까지 덧붙이면 혼동은 더 커진다.

- VPN이 TLS를 쓰니 내부 HTTPS는 중요하지 않다고 오해한다
- 443 포트를 쓰니 HTTPS와 같은 성질이라고 오해한다
- mTLS가 있으니 HTTP 요청 주체도 충분히 식별된다고 오해한다

즉 문제의 핵심은 보안 기능이 부족해서가 아니라, **서로 다른 계층의 책임과 암호화 객체가 섞여 있다는 점**에 가깝다.

## As-Is와 To-Be

아래 그림은 이 글에서 비교하려는 구조를 먼저 요약한 것이다.

![VPN 인증과 HTTP 인증의 as-is / to-be 비교](/assets/img/posts/2026/vpn-http-auth-as-is-to-be.svg){: w="1100" }
_As-is는 VPN 접속과 내부 서비스 신뢰가 느슨하게 결합되어 있고, to-be는 네트워크 진입 인증과 HTTP 요청 인증을 명시적으로 분리한다._

### As-Is

as-is에서는 보통 이런 생각이 숨어 있다.

- VPN에 접속했으니 내부 사용자다
- 내부망 트래픽이니 애플리케이션 인증은 약해도 된다
- HTTPS는 쓰고 있지만, 왜 쓰는지와 어디까지 보호하는지는 명확하지 않다

이 모델은 초기에 단순해 보이지만, 시간이 갈수록 권한 분리와 감사 추적이 어려워진다.  
특히 Grafana, 백오피스, 운영 API처럼 민감한 내부 서비스가 늘어날수록 "망 안에 들어온 사람"과 "이 작업을 수행할 권한이 있는 사람"을 같은 집합으로 보기 어려워진다.

### To-Be

to-be에서는 책임을 다음처럼 나눈다.

1. VPN의 mTLS는 사설망 진입 가능 여부를 결정한다.
2. 내부 HTTPS는 애플리케이션 구간의 전송을 보호한다.
3. 내부 HTTP 서비스의 인증 체계는 요청 주체와 권한을 별도로 검증한다.

즉 to-be는 "보안을 더 많이 붙인다"기보다 **어느 계층이 무엇을 보장하는지 명시적으로 분리한다**는 의미가 더 크다.

## 먼저 구분해야 할 암호화 객체

이 글을 정리하면서 가장 먼저 분리한 것은 "무엇이 암호화되고 무엇이 검증되는가"였다.

| 계층 | 보호/검증 대상 | 핵심 질문 |
| --- | --- | --- |
| VPN mTLS | 터널 양 끝 주체의 신원, 인증서 체인, 개인 키 소유 | 누가 터널을 열 수 있는가 |
| VPN 터널 암호화 | 사설망으로 흐르는 패킷 | 공용 네트워크 구간에서 무엇을 숨기고 변조를 막는가 |
| 내부 HTTPS | HTTP 요청/응답 메시지, 서비스 신원 | 어떤 애플리케이션 메시지를 보호하는가 |
| HTTP 인증/권한 | 사용자 세션, 토큰, 역할, 권한 | 이 요청을 실제로 수행할 수 있는가 |

이 구분을 먼저 세우면, 같은 TLS를 쓰더라도 역할이 서로 다르다는 점이 선명해진다.

## 통신 경로 관점에서 보면 무엇이 달라지는가

통신 경로 기준으로 보면 as-is와 to-be의 차이는 아래처럼 요약할 수 있다.

| 관점 | As-Is | To-Be |
| --- | --- | --- |
| 네트워크 진입 | VPN 접속 성공 = 내부 신뢰로 간주 | VPN 접속 성공 = 사설망 경로 획득으로만 간주 |
| 요청 흐름 | 네트워크 경계와 서비스 경계가 사실상 합쳐짐 | 네트워크 경계와 서비스 경계를 분리 |
| 운영 판단 | "망 안에 있으니 허용" | "망 안에 들어왔지만 요청 권한은 다시 확인" |
| 오프보딩 | VPN 회수와 서비스 계정 회수가 느슨하게 연결 | VPN 인증서 폐기와 서비스 권한 회수를 별도 절차로 관리 |

조금 더 풀어 쓰면 이렇다.

### As-Is의 통신 흐름

1. 사용자가 VPN에 접속한다.
2. 사설 IP와 내부 DNS 해석 경로를 획득한다.
3. 내부 ALB, Grafana, 백오피스 등으로 바로 접근한다.
4. 일부 서비스는 로그인하지만, 일부 서비스는 네트워크 위치만으로 신뢰될 수 있다.

이 흐름의 문제는 `3`과 `4` 사이의 경계가 불명확하다는 점이다.  
접속은 성공했지만, 그 다음 요청이 어떤 인증 단계를 거쳐야 하는지가 서비스마다 다르면 운영 기준이 흔들린다.

### To-Be의 통신 흐름

1. 사용자가 mTLS로 VPN Endpoint에 접속한다.
2. 검증이 끝나면 사설망 경로만 열린다.
3. 사용자는 내부 HTTPS 서비스에 요청을 보낸다.
4. 각 서비스는 `Authorization`, 세션 쿠키, OIDC/OAuth 2.0, SSO 등 자체 인증 체계로 요청을 다시 검증한다.
5. 애플리케이션 권한은 RBAC, 조직 권한, 역할 정책으로 별도 판단한다.

즉 VPN은 라우팅과 네트워크 경계를 다루고, HTTP 서비스는 요청 인증과 권한을 다룬다.

## AWS Client VPN의 mTLS는 무엇을 인증하는가

AWS Client VPN의 Mutual TLS는 요청 본문이나 HTTP 헤더를 인증하지 않는다.  
인증 대상은 **터널 양 끝의 주체, 신뢰 체인, 그리고 개인 키 소유 여부**다.

연결 시점의 흐름을 간단히 적으면 아래와 같다.

1. 클라이언트는 서버 인증서를 받고, 자신이 신뢰하는 CA가 서명했는지 확인한다.
2. 서버는 클라이언트 인증서를 받고, 자신이 신뢰하는 CA 체인에 속하는지 확인한다.
3. 양쪽은 각자 개인 키를 직접 드러내지 않고, 핸드셰이크 서명으로 키 소유를 증명한다.
4. 검증이 끝나면 세션 키를 합의하고, 그 키로 VPN 제어 채널과 데이터 채널을 보호한다.

이 단계에서 확인되는 것은 아래 두 가지다.

- 내가 접속한 VPN Endpoint가 우리가 신뢰하는 대상인가
- 접속한 사용자가 우리가 발급한 클라이언트 인증서를 가진 주체인가

즉 여기서 얻는 결과는 "이 단말은 사설망 진입 권한이 있다"이지,  
"이 사용자는 Grafana의 Admin 권한이 있다"가 아니다.

여기서 특히 중요한 점은 **인증서 자체가 사용자를 증명하는 것이 아니라, 인증서에 대응하는 개인 키를 실제로 가진 주체가 증명된다**는 점이다.  
인증서는 공개 가능한 신원 문서에 가깝고, 진짜 비밀은 개인 키다.  
따라서 mTLS의 본질은 "증명서를 보여줬다"보다 "그 증명서의 개인 키를 가진 주체가 실제로 응답에 서명했다"에 더 가깝다.

## VPN TLS에서 실제로 보호되는 것은 무엇인가

암호화 관점에서 보면 VPN TLS는 보통 두 역할을 동시에 수행한다.

1. 신뢰 가능한 상대와만 세션을 만든다.
2. 세션이 성립한 뒤 오가는 패킷을 대칭키로 보호한다.

첫 단계는 공개키 기반 구조(PKI)의 영역이다.  
CA가 인증서에 서명하고, 클라이언트와 서버는 그 서명을 검증해 서로의 신원을 확인한다.  
두 번째 단계는 세션 키의 영역이다.  
핸드셰이크가 끝나면 이후 트래픽은 공개키로 계속 암호화되는 것이 아니라, 협상된 대칭 세션 키로 보호된다.

이 점을 분리해서 보면 VPN TLS는 아래 순서로 이해하는 편이 정확하다.

- 인증서 체인 검증: 어떤 공개키를 신뢰할지 결정
- 개인 키 소유 증명: 인증서의 주체가 실제 참여자임을 확인
- 세션 키 합의: 이후 트래픽에 사용할 대칭키 생성
- 터널 암호화: 실제 패킷을 세션 키로 기밀성/무결성 보호

즉 "TLS를 쓴다"는 말 안에는 이미  
**신원 검증 단계와 데이터 보호 단계가 함께 들어 있다.**

## 암호화 관점에서 보면 네 층이 분리된다

아래 그림은 인증과 암호화 책임을 계층별로 나눈 모습이다.

![VPN과 HTTP의 계층별 인증 및 암호화 흐름](/assets/img/posts/2026/vpn-http-auth-layered-controls.svg){: w="1100" }
_인증서 체인 검증, 터널 암호화, 내부 HTTPS, HTTP 요청 인증은 서로 연결되지만 서로 다른 객체를 보호한다._

암호화 관점에서 가장 중요한 것은 "TLS를 쓴다"는 한 문장으로 모든 보호 범위를 설명하지 않는 것이다.

### 1. 터널 수립 단계의 TLS

이 단계의 목적은 상대방 신원 확인과 안전한 키 합의다.  
클라이언트와 서버는 인증서 체인과 개인 키 소유를 확인한 뒤 세션 키를 만들고, 그 뒤에 VPN 터널을 성립시킨다.

보호 대상:

- VPN Endpoint의 신원
- 클라이언트 인증서 소유 주체
- 터널을 여는 핸드셰이크 제어 채널
- 세션 키 합의 결과의 정당성

### 2. 터널 내부 트래픽 보호

터널이 성립한 뒤에는 내부 DNS 질의, 사설 IP 통신, 내부 ALB 접근 같은 패킷이 암호화된 터널 안을 지나간다.  
즉 공용 네트워크 경로에서 평문 노출이나 변조 가능성을 줄인다.

보호 대상:

- 사설망으로 전달되는 패킷
- 외부 네트워크 구간에서의 도청 및 변조 위험

### 3. 애플리케이션 자체의 HTTPS

VPN에 들어왔다고 해서 내부 서비스 HTTPS가 불필요해지는 것은 아니다.  
오히려 내부 ALB와 운영용 웹 서비스도 HTTPS를 유지하는 편이 역할을 더 명확하게 만든다.  
여기서 내부 HTTPS는 VPN 핸드셰이크와는 **별개의 TLS 세션**이다.

보호 대상:

- 애플리케이션 계층의 요청과 응답
- 내부 구간에서의 서비스 신원 검증
- TLS 종료 지점 이후를 제외한 HTTP 메시지의 기밀성 및 무결성

즉 요약하면 이렇다.

- mTLS는 **누가 터널을 여는가**를 검증한다
- VPN 암호화는 **어떤 패킷이 공용 구간을 안전하게 지나는가**를 다룬다
- HTTPS는 **어떤 애플리케이션 메시지를 보호하는가**를 다룬다

## 내부 HTTPS는 VPN 위에 올라가지만 VPN과 같은 것은 아니다

이 지점도 자주 헷갈린다.  
사용자 입장에서는 "VPN 켜고 브라우저로 `https://internal.example.com` 접속" 한 번으로 보이지만, 내부적으로는 서로 다른 보호 층이 겹쳐져 있다.

1. 먼저 VPN TLS가 세션을 만들고 사설 IP 경로를 연다.
2. 그 위에서 브라우저와 내부 ALB/서비스가 또 별도의 HTTPS 세션을 만든다.
3. HTTP 쿠키나 `Authorization` 헤더는 그 HTTPS 세션 안에서 전송된다.

즉 내부 요청은 개념적으로 보면 "VPN으로 보호된 네트워크 경로 위를, 다시 HTTPS로 보호된 HTTP 메시지가 지나가는 구조"다.  
중요한 것은 중복 여부가 아니라 **보호 대상이 서로 다르다**는 점이다.

- VPN은 네트워크 경로와 패킷 운반을 보호한다
- HTTPS는 애플리케이션 메시지와 서비스 신원을 보호한다
- HTTP 인증은 그 메시지를 보낸 주체와 권한을 판단한다

그래서 "VPN이 있으니 HTTPS를 빼도 된다" 혹은 "HTTPS가 있으니 VPN 인증만으로 충분하다" 같은 말은 각각 다른 책임을 서로 대체하려는 표현에 가깝다.

## 443 포트를 쓴다고 해서 곧바로 HTTPS인 것은 아니다

AWS Client VPN이 `443/UDP`를 사용하는 경우, 얼핏 보면 HTTPS와 비슷하게 느껴질 수 있다.  
하지만 포트 번호가 같다고 프로토콜 계층의 의미가 같아지는 것은 아니다.

구분해야 할 포인트는 아래와 같다.

- HTTPS는 HTTP 메시지를 TLS 위에 올리는 애플리케이션 프로토콜이다
- VPN은 사설망 트래픽을 운반하기 위한 터널링 계층이다
- 둘 다 TLS를 활용할 수 있지만, 캡슐화하는 데이터와 보장하려는 대상이 다르다

즉 `443/UDP`는 운영적으로는 방화벽 친화적인 선택일 수 있지만,  
그 사실만으로 HTTP 인증이나 애플리케이션 보안 설계가 자동으로 해결되지는 않는다.

## HTTP 표준 관점에서 보면 인증은 메시지 안에서 정의된다

이 지점을 분리하는 데 가장 도움이 된 것은 HTTP 표준 관점이었다.  
HTTP Semantics(RFC 9110) 기준으로 인증은 결국 요청과 응답 메시지 수준에서 표현된다.

대표적으로는 아래 요소들이 있다.

- 클라이언트의 `Authorization` 헤더
- 서버의 `401 Unauthorized`
- 서버가 인증 방식을 제시하는 `WWW-Authenticate`
- 프록시 인증의 경우 `407 Proxy Authentication Required`
- 세션 쿠키, OIDC, OAuth 2.0 같은 상위 인증 체계

이 관점에서 보면 VPN의 mTLS는 HTTP 바깥에서 먼저 동작한다.  
즉 아직 `GET /dashboard`나 `POST /admin/users` 같은 HTTP 요청이 도착하기도 전에, 터널 자체를 열지 말지를 판단하는 절차다.

그래서 다음 문장은 엄밀히 보면 서로 다른 뜻을 가진다.

- "이 사용자는 VPN에 접속할 수 있다"
- "이 사용자는 이 HTTP 요청을 수행할 수 있다"

첫 번째 문장은 네트워크 경계에 관한 말이고,  
두 번째 문장은 애플리케이션 행위와 권한에 관한 말이다.  
둘을 같은 문장처럼 다루면 설계가 단순해지는 것이 아니라 오히려 모호해진다.

## 운영 관점에서 .ovpn은 설정 파일이 아니라 자격 증명 번들이다

`.ovpn` 파일은 겉보기에는 단순한 클라이언트 설정처럼 보인다.  
하지만 실제로는 클라이언트 인증서, 개인 키, CA 정보가 함께 포함될 수 있다.  
운영적으로 보면 평범한 설정 파일이 아니라 **민감한 자격 증명 번들**에 더 가깝다.

그래서 온보딩 절차는 단순 설치 가이드보다 더 넓게 봐야 한다.

- 어떤 채널로 전달할 것인가
- 로컬 디스크에 어떻게 저장할 것인가
- 재발급 시 이전 인증서를 어떻게 폐기할 것인가
- 퇴사나 역할 변경 시 어떤 순서로 회수할 것인가

기술적으로 mTLS를 붙였더라도, 자격 증명 전달과 저장 절차가 느슨하면 전체 보안 수준은 그만큼 낮아진다.

## CRL은 발급의 반대편에 있는 핵심 운영 절차다

인증서 기반 모델은 발급보다 폐기 절차에서 더 자주 실패한다.  
AWS Client VPN에서도 오프보딩의 핵심은 결국 CRL 관리다.

CRL은 이미 발급된 인증서 중 더 이상 신뢰하면 안 되는 대상을 기록한 목록이다.  
즉 "과거에 발급했다"와 "지금도 유효하다"를 분리하는 장치다.

이 관점에서 보면 오프보딩은 단순 계정 삭제가 아니라 아래 순서에 가깝다.

1. 대상 클라이언트 인증서를 폐기한다.
2. 폐기 이력을 반영한 CRL을 다시 생성한다.
3. VPN Endpoint가 최신 CRL을 참조하도록 반영한다.
4. 내부 서비스 계정과 역할도 별도로 회수한다.

여기서 중요한 점은 CRL이 보통 누적 관리된다는 것이다.  
즉 마지막 사용자 한 명만 반영하는 것이 아니라, 지금까지 폐기된 인증서 전체 집합이 반영되어야 한다.

## To-Be 설계 기준

이 글을 정리하고 나서, 내 기준의 to-be는 아래처럼 표현할 수 있게 됐다.

1. VPN은 네트워크 진입 가능 여부만 보장한다.
2. 내부 HTTPS는 애플리케이션 통신 구간을 보호한다.
3. HTTP 서비스는 자체 인증과 권한 모델을 유지한다.
4. VPN 자격 증명과 서비스 계정은 별도 수명 주기로 관리한다.
5. 감사 로그도 네트워크 접속 로그와 애플리케이션 행위 로그를 분리해 본다.

결국 핵심은 VPN을 과대평가하지 않는 것이다.  
VPN은 매우 중요한 보안 계층이지만, 그것만으로 HTTP 요청 인증과 애플리케이션 권한 모델을 대체하지는 않는다.  
정확히는 **사설망 진입 경로를 통제하는 계층**이고, 그 위의 HTTP와 애플리케이션 보안은 여전히 별도로 설계되어야 한다.

이 기준이 생기고 나니, 앞으로는 내부 서비스 인증 구조를 볼 때도 "무엇이 무엇을 보호하는가"를 더 선명하게 나눠서 볼 수 있게 됐다.  
특히 다음에는 내부 ALB에서의 TLS 종료 지점, 백엔드 간 mTLS 필요성, SSO와 VPN 조합의 감사 추적 모델까지 더 이어서 정리해볼 생각이다.
